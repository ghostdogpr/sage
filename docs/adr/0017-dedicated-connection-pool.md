# Dedicated connection pool: disposable slots, bounded wait, liveness-gated

Blocking commands borrow a Dedicated Connection from an on-demand pool, run alone on it, and release it. The pool's behavior is shaped by four decisions, each refining an earlier connection-model ADR.

**Disposable slots, not self-reconnecting connections.** A Dedicated Connection is a plain pooled resource: created on demand (running the same HELLO bootstrap as any connection), returned to an idle set on release, and discarded outright on any connection loss or error. It carries no reconnect loop — auto-reconnect stays solely in the Multiplexed Connection (ADR-0003). A reconnect loop here would serve nothing: a blocking command whose connection drops has already failed `ConnectionLost(mayHaveExecuted = true)` per ADR-0006 (the `BLPOP` may have popped before the reply was lost), so reconnecting the socket underneath it is meaningless. The next acquire reuses an idle connection or establishes a fresh one.

**Bounded, with a waiting acquire.** The pool has a configurable max (`maxConnections`, default 8). On exhaustion the caller parks — as a fiber, not a thread — up to `acquireTimeout` (default 5s), then fails `TimedOut`. The bound caps connections against the server's `maxclients`; the wait absorbs normal contention bursts instead of failing them. `acquireTimeout` governs only the wait for a slot and is unrelated to a blocking command's own server-side timeout, which may legitimately be `Forever`. The rejected alternatives: unbounded creation (a runaway fiber count opens sockets until the server rejects them — a clean failure, but late and after resource damage) and fail-fast-on-exhaustion (surprising for blocking workloads where brief contention is normal, and it would need a new sealed error case).

**Gated on client liveness.** A blocking command issued while the Multiplexed Connection is not `Live` fails fast `NotConnected`, rather than attempting its own connect. This keeps one coherent connectivity signal for the whole client and matches ADR-0006's contract for ordinary commands submitted while disconnected; the alternative — an independent pool that always tries to connect — makes every blocking call eat a `connectTimeout` against a down server and fail with a different error than ordinary commands do.

**Prompt force-close of blocking commands on close.** ADR-0013 drains in-flight replies up to `closeTimeout` so already-sent commands collect their *imminent* replies instead of a pessimistic `ConnectionLost(true)`. A parked blocking command is the exception its rationale does not cover: a `blPop(Forever, …)` has no imminent reply, so waiting `closeTimeout` for it would routinely stall shutdown whenever a consumer fiber is parked. On `close`, therefore, idle pooled connections close at once and in-flight blocking commands are force-closed immediately → `ConnectionLost(true)`; the Multiplexed Connection's drain is unchanged. This honors ADR-0013's reasoning (drain is for imminent replies) rather than its letter.

## Consequences

- No watchdog runs on Dedicated Connections, and connections are not validated on borrow. Idle connections are evicted after `idleTimeout` (default 30s), bounding staleness and returning sockets when blocking load subsides; a hung call on a half-open survivor is the user's deadline to own, per ADR-0015.
- New `SageConfig` surface: a dedicated-pool config carrying `maxConnections`, `acquireTimeout`, and `idleTimeout`.
- The pool reads the Multiplexed Connection's liveness state, so it holds a reference to it rather than standing fully independent. The Multiplexed Connection is the sole authority on its Generation (the epoch a Dedicated Connection is stamped with): it answers `liveGeneration` (the epoch to stamp, captured atomically and only while live) and `isCurrent` (whether a stamp is still current), so the pool never compares raw epoch numbers itself. Both are liveness-gated: the epoch only bumps when the *next* socket goes live, so a bare epoch match would read as current throughout a reconnect window — `isCurrent` therefore also requires `Live`, so an idle connection is never reused when a reconnect races an in-flight acquire that passed the liveness gate a moment earlier. Stamping at the moment a connection joins the pool — not when its socket opens — means one built across a reconnect is born current rather than discarded-and-retried.

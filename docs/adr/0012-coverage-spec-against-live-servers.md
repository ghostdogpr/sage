# Coverage Spec diffs against live servers, not vendored specs

The Coverage Spec's source of truth is the command list reported by the running Redis and Valkey containers (`COMMAND LIST`/`COMMAND DOCS`), not vendored `commands.json` fixtures as ADR-0005 originally sketched. The integration suite already boots those exact server versions, so the spec can never drift from the tested binary, and bumping the container image *is* the spec update — new commands in a new release immediately fail as unacknowledged TODOs. The vendored alternative (checked-in fixtures plus an update script) was rejected because the pin and the test image are two things to keep in sync, and a stale fixture silently reports yesterday's coverage. The cost: the coverage check needs Docker, so it lives in `integration-tests`.

The implemented side of the diff comes from per-family sample fixtures in core's test scope: one constructed `Command` per builder variant, reaching every distinct wire name. A builder without a sample shows up as a coverage gap, so the list cannot silently under-count; the same samples serve as golden encode assertions. Deliberate skips (`Coverage.skipped`, with reasons) and acknowledged gaps (`Coverage.todo`) are Scala constants beside the suite; the partition must be exact — a spec command in no bucket, a stale skip/todo entry, or an implemented name unknown to both servers each fail with a diff. Missing-but-acknowledged never fails: completeness is reported per server, not gated.

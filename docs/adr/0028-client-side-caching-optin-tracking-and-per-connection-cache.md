# Client-side caching: opt-in tracking with a per-generation cache

A Cached Read opts a single read into client-side caching with a TTL (`client.cached(command, ttl)`); the result is served from a local cache until the server says it changed or the TTL lapses. The server side is RESP3 client tracking, which has three modes, and the choice is forced by sage auto-pipelining every fibre's commands onto one Multiplexed Connection.

Tracking is enabled in **OPTIN** mode (`CLIENT TRACKING ON OPTIN`) at the connection bootstrap when `CacheConfig.enabled` (the default), and each cached read is sent as the adjacent pair `[CLIENT CACHING YES, <read>]` — one atomic batch on the wire, the control reply discarded, exactly as `ASKING` already prefixes a redirected command. The alternatives both fit the multiplexed connection badly: default tracking would have the server remember *every* key read by *any* fibre (the firehose of all normal reads) and push invalidations for keys nobody chose to cache, and BCAST would stream an invalidation for every write in the whole keyspace down the command connection. OPTIN keeps both the server's tracking table and the invalidation traffic proportional to what was actually cached. The classic objection to OPTIN — that `CLIENT CACHING YES` must immediately precede its command, which a shared connection cannot guarantee — does not apply here because the connection can already enqueue an atomic adjacent pair under its lock. No `REDIRECT` and no `CLIENT ID` are needed: RESP3 delivers invalidation pushes on the same connection that enabled tracking. `NOLOOP` is deliberately *not* set, so the client's own write to a key it cached produces a self-invalidation that evicts the entry — read-your-writes consistency with no coupling between the write path and the cache.

The cache is owned **per connection generation**, not once per client. The Multiplexed Connection already builds a fresh state object per generation so a dead generation's late frames can never misalign the live one; the cache rides the same structure, which makes the "reconnect flushes the cache" requirement (tracking state is connection-bound) fall out for free — a reconnect discards the generation and its cache, the new generation starts empty, and a late invalidation push from the dying socket lands on the dying cache, never the new one. An entry is keyed by the command's full wire bytes (so `GET k`, `GETRANGE k …` and `MGET k …` are distinct) and stores the raw reply Frame, re-decoded per hit through the same path a live reply takes; a reverse index from each tracked key (by content equality) to the entries that touched it turns an invalidation push into an eviction. Cacheability is a property distinct from read-only: `Command` carries both an `isReadOnly` bit (side-effect-free, a future input to replica routing) and a narrower `cacheable` bit, set by the builders. Cacheable means the result is a pure function of the named keys' current state, so an invalidation push covers every way it can change — `GET`, `HGETALL`, `ZRANGE`. Reads that are read-only but **not** cacheable are built with `Command.readUncacheable`: time-varying ones (`TTL`, `OBJECT IDLETIME`) and non-deterministic ones (`SRANDMEMBER`, `RANDOMKEY`) change with no key write, so no invalidation would ever fire and caching them would freeze a stale value. `cached` requires a cacheable command with at least one key; anything else (a write, a keyless read, a `readUncacheable` read) fails with `NotCacheable` rather than silently going stale. Concurrent misses for the same command are single-flighted onto one server read. The cache is bounded in bytes with LRU eviction (`CacheConfig.maxBytes`), so a single large value cannot blow the budget, and TTL is absolute, checked lazily on lookup.

On a cluster client `cached` runs the read **without** caching, so the same call stays portable between standalone and cluster (PRD topology portability). Per-node tracking through redirects and failover is a deliberate follow-up; the per-generation cache design already accommodates it (each node connection would own its own).

## Consequences

- Caching works only against RESP3 servers, which sage already requires (ADR-0002). Tracking is enabled at bootstrap by default because in OPTIN mode it costs nothing until a read is actually cached; `CacheConfig.enabled = false` skips it entirely (and makes `cached` run uncached) for environments where ACLs or a proxy permit `HELLO` but deny `CLIENT TRACKING`.
- A reconnect is a full cache flush. Hot caches go cold on every failover — the cost of binding tracking state to the connection rather than risking stale reads across a reconnect.
- The cache bound is per generation (per node in a cluster), not a single global budget; `CacheConfig.maxBytes` sizes each one.
- A cached read of a value larger than `maxBytes` still returns correctly but is evicted immediately, so it is never served from cache.
- Cluster client-side caching is not yet implemented; cached reads there are ordinary reads until that follow-up lands.
